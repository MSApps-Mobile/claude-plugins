# MSApps Recruiter — מערכת גיוס

Skill זה מכסה את כל תהליך הגיוס של MSApps — מפרסום משרות דרך הקפצת מודעות ועד סקירת מועמדים ומענה.

## מי אנחנו

MSApps היא חברת פיתוח תוכנה שמגייסת מפתחים ואנשי הייטק עבור לקוחות שונים (בעיקר תעשייה ביטחונית ופינטק). איש הקשר: **מיכל**, מייל: jobs@msapps.mobi, LinkedIn: www.linkedin.com/in/michalmsapps.

**כלל ברזל**: לעולם לא לחשוף את שם MSApps במודעות המשרה. המועמד לא צריך לדעת מי החברה המגייסת.

## כללי ברזל — תקשורת

### 🚫 אסור לחשוף מספר טלפון
**לעולם לא לכלול מספר טלפון (לא של מיכל ולא של MSApps) בשום מייל, הודעה, או תקשורת יוצאת** — לא לעומדדים, לא ללקוחות. ערוץ התקשורת היחיד הוא מייל: jobs@msapps.mobi.

### 📋 המסלול הנכון — תהליך גיוס שלב אחר שלב

**מסלול הגיוס הסטנדרטי** (לא לקפוץ שלבים!):

1. **סינון קו"ח** — בדיקת קורות החיים מול דרישות המשרה. אם המועמד לא עומד בדרישות — פסילה (מייל ידידותי).
2. **שאלות סינון** — שליחת שאלות מוקדמות (screening questions) למועמד שעבר את שלב ה-CV. ממתינים לתשובות.
3. **מבחן / בדיקת התאמה** — אם המשרה דורשת (קוד, ניתוח מערכות וכד'), שולחים מבחן. אם לא — ממשיכים ישירות.
4. **אישור לקוח** — מציגים את המועמד ללקוח (MSApps Client) לפני שמתקדמים. רק לאחר שהלקוח מאשר שהוא רוצה להמשיך — מתקדמים.
5. **ראיון/שיחה** — רק בשלב האחרון, לאחר אישור הלקוח, מתאמים ראיון עם המועמד.

**עיקרון מנחה**: ממצים את התקשורת הכתובה עד תום לפני שמגיעים לשיחה. שיחה/ראיון הם המשאב הכי יקר — שומרים אותם לשלב שבו ברור שיש התאמה מצד כולם. לא מציעים שיחה בשום שלב לפני שהלקוח אישר את המועמד.

### 📨 מסלול LinkedIn Outreach (InMail) — כשפנינו למועמד פרואקטיבית

כאשר פנינו למועמד ביוזמתנו דרך LinkedIn/Sales Navigator והוא הביע עניין:

1. **מועמד מגיב בחיוב** → מבקשים קורות חיים. **לא מזמינים לשיחה!**
2. **מקבלים קו"ח** → בודקים התאמה מול דרישות המשרה.
3. **אם מתאים** → מעבירים ללקוח לאישור.
4. **לקוח מאשר** → רק אז מתאמים ראיון/שיחה.

**הנוסח הנכון לאחר שמועמד מביע עניין ב-LinkedIn:**
> "Great to hear! I'd love to learn more about your background. Could you send over your CV to jobs@msapps.mobi (please mention MSA-XXX in the subject line)? I'll review it and get back to you."

### ❌ אסור לקפוץ ישר לשיחה — בשום שלב
לעולם לא להציע שיחה, ראיון, או "call" לפני שכל התנאים הבאים מתקיימים: (א) קיבלנו קורות חיים, (ב) בדקנו התאמה, (ג) הלקוח אישר. כלל זה חל על כל סוגי המועמדים — אלה שפנו אלינו ואלה שפנינו אליהם.

**דוגמאות שגויות** — כזה אסור לשלוח:
> "...We would like to move forward and schedule an initial call to learn more about your background..."

> "...Would you be open to a short call this week to learn more? Happy to work around your schedule..."

מה רע בדוגמאות: מזמינות לשיחה לפני שראינו קו"ח ולפני אישור לקוח.

### סינון לפני פנייה ⚠️
תמיד לסנן מועמדים קודם — לבדוק את קורות החיים מול דרישות המשרה (שנות ניסיון, טכנולוגיות, רקע נדרש) — ורק אחרי שהמועמד נמצא מתאים, לפנות אליו. הסיבה: פנייה למועמד שלא עומד בדרישות מבזבזת זמן של שני הצדדים ופוגעת במקצועיות. הסדר הנכון הוא: קריאת קו"ח → בדיקת התאמה לדרישות → שאלות סינון → (מבחן אם רלוונטי) → אישור לקוח → רק אז שיחה.

### מיילים למועמדים ✅

#### ⚠️ כלל ברזל — שיטת שליחה בלעדית (מאוד חשוב!)

**אסור בהחלט להשתמש ב-Gmail MCP (`gmail_create_draft`) לצורך שליחת מיילים למועמדים.**

הסיבה: Gmail MCP פועל על החשבון `msmobileapps@gmail.com` — הג'ימייל האישי של מיכל. שליחה משם במקום מ-`jobs@msapps.mobi` אינה מקצועית ומבלבלת את המועמדים.

**הדרך היחידה לשלוח מיילים למועמדים — דרך הדפדפן:**
1. פתח את הדפדפן דרך Claude in Chrome (`tabs_context_mcp` → `navigate`)
2. נווט ל-Gmail: `https://mail.google.com` (תוך כדי מחובר כ-jobs@msapps.mobi)
3. כתוב ושלח את המייל משם — כך ה-FROM יהיה תמיד `jobs@msapps.mobi`

**Gmail MCP מותר רק לקריאה** — לקרוא/לחפש מיילים נכנסים (label: `Jobs-MSApps`). אסור להשתמש בו לשליחה או יצירת טיוטות למועמדים.

מיילים למועמדים (אישור קבלה, שאלות סינון, דחייה מנומסת) אינם דורשים אישור מיכל מראש — למעט הזמנה לשיחה (ראה מסלול הגיוס למעלה).

### מיילים ללקוחות ⛔
1. **אסור לשלוח מיילים ללקוחות ישירות** — בכל מקרה שרוצים לשלוח מייל ללקוח (פרופיל מועמד, עדכון, כל דבר), מכינים טיוטה ושולחים אותה למיכל בכתובת msmobileapps@gmail.com. מיכל מחליטה אם ומתי להעביר.
2. **אסור לכלול ציפיות שכר במיילים ללקוחות** — ציפיות שכר הן מידע פנימי בלבד. לעולם לא מופיעות בתקשורת מול לקוח.
3. **אסור לבדות מועמדים** — לשלוח ללקוח רק מועמדים שקיימים בגיליון מעקב המועמדים עם פרטים מלאים ומאומתים. אין ליצור פרופילים מדומים.
4. **חובה לתעד לפני לשלוח** — לפני כל שליחת פרופיל ללקוח, המועמד חייב להיות רשום בגיליון מעקב המועמדים.

**שלב 2 — פרסום בפלטפורמות**
פרסם את המשרה בכל הפלטפורמות החינמיות הזמינות. ראה את הפירוט בקובץ `references/platforms.md`.

**שלב 3 — עדכון הגיליון**
לאחר פרסום, עדכן בגיליון באילו פלטפורמות המשרה פורסמה ואת הסטטוס.

### הגדרות קבועות לכל פרסום

כל מודעת משרה חייבת לכלול:
- **מספר משרה**: MSA-XXX (לפי הגיליון) — מופיע בתחילת המודעה

- **איש קשר**: מיכל
- **בקשה מהמועמד**: "נא לציין את מספר המשרה בנושא המייל"

- **קטגוריה**: הייטק וטכנולוגיה
- **מסלול פרסום**: תמיד חינמי

### עדכון/עריכת מודעה קיימת ב-Zuze

## חלק 2: הקפצת מודעות (Ad Bumping)

הקפצה היא פעולה יומית שמעלה את המודעות שלנו לראש הלוח ב-Zuze, כדי שיהיו חשופות יותר.

### תנאי מקדים — התחברות

צריך להיות מחובר לחשבון Zuze. אם מופיע "אורח יקר" — המשתמשת צריכה להתחבר ידנית דרך דף הבית zuze.co.il.

### תהליך ההקפצה — אוטומציה via JavaScript fetch()

דף רשימת המודעות: `https://www.zuze.co.il/listpost.php?id_ad=1615901880&id_kat1=42`

הדף מציג 20 מודעות בכל עמוד, עם pagination בתחתית (עמודים 1, 2, 3, 4...). כרגע יש 73 מודעות = 4 עמודים.

**מבנה הדף**: כל מודעה מכילה אנקור נסתר `<a>` עם הטקסט "הקפצת מודעה" ו-href ייחודי. הקישורים קיימים ב-DOM גם בלי לפתוח את סקשן "לפרטים", כלומר אפשר לאסוף אותם ישירות.

**שלב 1 — איסוף קישורי הקפצה והרצה (לכל עמוד)**:
```javascript
const allLinks = document.querySelectorAll('a');
const bumpHrefs = [];
allLinks.forEach(link => {
  if (link.textContent.includes('הקפצת מודעה') && link.href) {
    bumpHrefs.push(link.href);
  }
});

async function bumpAds(urls) {
  const results = [];
  for (let i = 0; i < urls.length; i++) {
    try {
      const resp = await fetch(urls[i], { credentials: 'include' });
      results.push({index: i, status: resp.status, ok: resp.ok});
    } catch (e) {
      results.push({index: i, error: e.message});
    }
  }
  return results;
}

window._bumpResultsN = null;
bumpAds(bumpHrefs).then(r => window._bumpResultsN = r);
```
החלף `N` במספר העמוד (1, 2, 3, 4...).

**שלב 2 — המתנה לתוצאות** (~15-20 שניות לכל עמוד של 20 מודעות):
```javascript
window._bumpResultsN ? JSON.stringify({
  count: window._bumpResultsN.length,
  allOk: window._bumpResultsN.every(r => r.ok),
  statuses: window._bumpResultsN.map(r => r.status)
}) : 'not ready yet'
```
אם מחזיר 'not ready yet' — חכה עוד 10-15 שניות ונסה שוב.

**שלב 3 — ניווט לעמוד הבא** (חזור על שלבים 1-2 לכל עמוד):
```javascript
const links = document.querySelectorAll('a');
for (const link of links) {
  if (link.textContent.trim() === '2') { // או '3', '4' וכו'
    link.click();
    break;
  }
}
```
חכה 3 שניות אחרי הקליק לטעינת העמוד, ואז חזור לשלב 1.

### טיפים חשובים

- **credentials: 'include'** — חובה! בלי זה ה-fetch לא שולח את העוגיות של Zuze והבקשה נכשלת.
- **הרצה סדרתית** (for loop, לא Promise.all) — כדי לא להעמיס על השרת.
- כל בקשה מוצלחת מחזירה **status 200**.
- **אל תחזיר את ה-URLs עצמם בתוצאות** — הם מכילים query strings שעלולים להיחסם ע"י ה-extension. החזר רק count/status.
- **אם ה-Chrome extension מתנתק** בזמן ההמתנה — אל דאגה, ה-fetch() ממשיך לרוץ בדפדפן ברקע. פשוט חכה ובדוק שוב את `window._bumpResultsN`.

### תזמון

ההקפצה רצה **פעם ביום** כחלק מה-scheduled task `job-posting-automation` (14:03). כדאי שהמשתמשת תהיה מחוברת ל-Zuze לפני ההרצה.

## חלק 3: סקירת מועמדים (Review Job Applications)

משימה זו כבר רצה כ-scheduled task (`review-job-apllications`) בכל יום ב-10:00. תפקידה:

### מה המשימה עושה

1. **קריאת מיילים** — בודקת מיילים חדשים ב-jobs@msapps.mobi (Gmail, לייבל Jobs-MSApps) עם קורות חיים
2. **זיהוי מספר משרה** — מזהה את מספר ה-MSA-XXX מנושא המייל או מגוף ההודעה
3. **הערכת התאמה** — סוקרת את קורות החיים ומעריכה התאמה למשרה המבוקשת
4. **תיעוד** — מעדכנת את [גיליון מעקב המועמדים](https://docs.google.com/spreadsheets/d/1v1U5t0OuWF7a2istDcdsOU9Kw7uQ04DiuE2A5UQRYJc/edit?gid=802645476) (tab "מעקב מועמדים") עם פרטי המועמד, סטטוס, והערות
5. **מענה** — שולחת תגובה למועמד ממייל `jobs@msapps.mobi` **דרך הדפדפן בלבד** (Claude in Chrome). אסור להשתמש ב-Gmail MCP לשליחה.

### טיפול במועמד שלא מתאים למשרה שפנה אליה

אם מועמד פנה למשרה X אבל המיומנויות שלו מתאימות יותר למשרה Y — מציעים לו את המשרה החלופית. דוגמה מתועדת: מועמדת פנתה ל-MSA-013 (AI) אבל הרקע שלה התאים ל-MSA-007 (DevOps) ו-MSA-008 (מנתח מערכות).

### עמודות בגיליון מעקב מועמדים

הגיליון כולל: מספר מועמד, מספר משרה (MSA-XXX), שם, מייל, טלפון, עיר, מקור (AllJobs/Zuze/ישיר), סטטוס, תאריך טיפול, הערות.
